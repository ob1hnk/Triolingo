# 편지 시스템 — 힌트 위장도(Hint Disguise) 평가 프레임워크

> 발표용 평가 자료. 부모 NPC가 생성한 답장 편지에 **다음 퀘스트 힌트
> (`next_quest_hint`)** 가 "부모의 추억담"처럼 얼마나 자연스럽게 위장되어
> 녹아들었는지를 **LLM-as-judge** 로 정량 평가한다.

---

## 1. 평가 목적

편지 시스템의 핵심 설계 의도는, 다음 퀘스트의 힌트를 **게임 지시문이 아니라
부모의 개인적 추억담처럼** 편지에 녹여 내는 것이다. 아이는 지금은 이것이
힌트인지 모르고, 나중에 퀘스트를 만난 뒤에야 "아, 그때 그 말이 이거였구나!"
하고 깨닫게 된다.

이 "위장이 얼마나 잘 되었는가"는 주관적으로 보이지만, **명확한 루브릭 + LLM
채점관**을 쓰면 일관되고 재현 가능한 점수로 측정할 수 있다. 본 프레임워크가
그 측정 도구다.

---

## 2. 평가 파이프라인

```
각 테스트 케이스마다
  ① 편지 생성  : 실제 운영 SYSTEM_PROMPT 그대로 사용 (gpt-4.1-mini, temp 0.7)
  ② 채점       : LLM-as-judge 가 1~5점 + 근거 출력  (gpt-4.1-mini, temp 0.0)
  ③ 수집       : 점수 / 위장 평가 / 누설 평가 / 근거
마지막에
  ④ 집계 + 시각화 : 케이스별 bar chart + 전체 평균선 (영어 그래프)
```

- **평가 대상 = 운영 대상**: 평가 스크립트는 별도 프롬프트를 만들지 않고,
  실제 코드의 `interaction/text/prompts/text_to_text_v1.py` 안의
  `SYSTEM_PROMPT` 를 그대로 import 해서 평가한다. → 평가 결과가 실제 게임
  동작을 그대로 반영한다.
- **생성 온도 0.7 / 채점 온도 0.0**: 생성은 운영과 동일한 다양성(0.7), 채점은
  재현성을 위해 결정론(0.0)으로 분리했다.

---

## 3. 평가 루브릭 (1~5점)

두 축으로 본다.
- **위장(Disguise)**: 힌트가 게임 지시문이 아니라 부모의 추억담/조언으로
  녹아들었는가?
- **정보 누설(Leakage)**: 도메인(예: "우물", "막힌 곳")만 언급하고, 정확한
  해결 단계는 노출하지 않았는가?

| 점수 | 기준 |
|:---:|---|
| **5** | 완벽한 위장. 전적으로 추억담 형태, 도메인만 언급, 해결 단계 무노출. 퀘스트를 만난 뒤에야 회상하며 깨닫는다. 메타 표현 없음. |
| **4** | 자연스러운 위장이나 사소한 흠. 힌트 의도가 살짝 비치거나 해결 방향을 너무 친절히 짚어 줌. |
| **3** | 위장 시도는 했으나 읽는 즉시 "이건 힌트구나" 하고 티가 남. 또는 해결 단계 일부가 드러남. |
| **2** | 약한 위장. 추억담이 형식적이고 해결 방법을 사실상 설명함. |
| **1** | 직접 노출 / 위장 실패. "다음 퀘스트에서 ~하세요" 식 게임 지시문, 도메인+해결 단계 평문 노출, 몰입 붕괴. |

---

## 4. LLM-as-judge 프롬프트

- **입력**: `next_quest_hint` + 생성된 편지
- **출력**: 아래 JSON (`response_format=json_object` 로 강제)

```json
{
  "score": 1~5,
  "disguise_quality": "위장(추억담화) 정도 한 줄 평",
  "leakage": "해결 단계 누설 여부 한 줄 평",
  "reason": "점수 근거 2~3문장"
}
```

채점관 프롬프트에는 위 루브릭 전문과, 좋은 위장/나쁜 위장의 정의를 명시해
채점 기준을 고정했다. (스크립트의 `JUDGE_SYSTEM_PROMPT` 참고)

---

## 5. 테스트 케이스 5개

서로 다른 도메인의 `next_quest_hint` × `child_letter` 조합. 각 케이스는 운영
프롬프트의 컨텍스트 포맷(`<letter_context>`)에 맞춰 구성된다.

| ID | 그래프 라벨 | 다음 퀘스트 힌트 (도메인) |
|---|---|---|
| TC1 | Blocked Well | 막힌 우물 — 막힌 곳 찾기 |
| TC2 | Dark Cave | 어두운 동굴 — 빛 밝히기 |
| TC3 | Broken Bridge | 무너진 다리 — 돌 쌓기 |
| TC4 | Locked Door | 잠긴 문 — 숨겨진 숫자 단서 |
| TC5 | River Crossing | 강 건너기 — 얕은 곳 찾기 |

---

## 6. 실행 방법

```bash
cd GoGoGolem/src/ai
.venv/bin/python ../../evaluation/letter_system/hint_disguise_eval.py
```

- API 키는 `GoGoGolem/src/ai/.env` 의 `OPENAI_API_KEY` 에서 자동 로드된다.
- 필요한 패키지(`openai`, `matplotlib`, `python-dotenv`)는 이미 설치돼 있다.

### 생성물

| 파일 | 내용 |
|---|---|
| `hint_disguise_scores.png` | 케이스별 점수 bar chart + 전체 평균선 (영어 그래프, 발표 슬라이드용) |
| `hint_disguise_results.json` | 생성된 편지 전문 + 점수 + 근거 (발표 부록/원자료용) |

> **참고**: 편지 생성 온도가 0.7 이라 실행할 때마다 점수가 조금씩 달라진다.
> 발표에는 한 번 돌린 결과의 그래프/JSON 을 그대로 캡처해 쓰면 된다.

## 7. 발표 슬라이드용 요약 멘트 (예시)

- **무엇을**: 편지에 다음 퀘스트 힌트가 "부모의 추억담"처럼 얼마나 자연스럽게
  위장됐는지를 1~5점으로 정량 평가.
- **어떻게**: 실제 운영 프롬프트로 편지 생성 → 동일 모델을 채점관(LLM-as-judge)
  으로 써서 루브릭 기반 채점 → 5개 도메인 케이스로 집계.
- **왜 신뢰할 수 있나**: ① 평가 대상이 실제 운영 프롬프트와 100% 동일,
  ② 채점은 온도 0 으로 결정론적, ③ 채점 기준(루브릭)을 프롬프트에 고정.
- **결과**: 케이스별 점수 + 전체 평균을 그래프로 제시 (그래프의
  `Overall average` 선).
